Kubernetes Operators Go

Building Self-Healing Infrastructure with Kubernetes Operators

Souvik· February 2026· 13 min read

A Kubernetes operator is how you teach Kubernetes about your application's operational domain knowledge. Instead of a human following a runbook to scale your database, restart a failed shard, or rotate credentials, an operator does it automatically based on observable cluster state. The pattern is elegant: you extend the Kubernetes API with a Custom Resource Definition, write a controller that reconciles the desired state with the actual state, and deploy it. Kubernetes becomes your automation platform, not just your container scheduler.

This post walks through writing a real operator using Kubebuilder — not a toy example, but the patterns I use in production operators. We'll cover the CRD, the reconciler, status conditions, leader election, and testing.

1. The CRD + Controller Pattern

A CustomResourceDefinition is just an extension of the Kubernetes API. You define a new resource kind (e.g., WebApp), its schema, and validation rules. Once the CRD is installed, users can create WebApp objects via kubectl apply just like they'd create a Deployment. The API server stores it in etcd, no different from any built-in resource.

The controller is a long-running process that watches for changes to your custom resources (and often to related built-in resources like Deployments and Services) via the Kubernetes watch API. When it detects a change, it runs the reconcile loop: compare desired state (the spec) with actual state (what's running in the cluster), and take whatever actions are necessary to close the gap. The reconciler is idempotent — it should be safe to run multiple times in succession with the same result.

Level-triggered, not edge-triggered: The reconciler is called with the resource's current state, not a diff of what changed. You don't get "the replica count changed from 3 to 5" — you get "here is the current WebApp object." Your reconciler must inspect the actual cluster state and take action accordingly. This is intentional: it makes recovery from controller crashes trivial, because the next reconcile loop handles everything regardless of what the previous one did or failed to do.

2. Scaffolding with Kubebuilder

bash — kubebuilder-scaffold.sh 
# Prerequisites: Go 1.22+, kubectl, kubebuilder v3
curl -L -o kubebuilder "https://go.kubebuilder.io/dl/latest/$(go env GOOS)/$(go env GOARCH)"
chmod +x kubebuilder && mv kubebuilder /usr/local/bin/

# Initialize the project (sets up go module, Makefile, controller-runtime)
mkdir webapp-operator && cd webapp-operator
kubebuilder init \
  --domain example.com \
  --repo github.com/yourorg/webapp-operator

# Create the API (generates CRD scaffold + controller scaffold)
kubebuilder create api \
  --group apps \
  --version v1alpha1 \
  --kind WebApp \
  --resource \
  --controller

# Generated structure:
# api/v1alpha1/webapp_types.go   — CRD spec/status types
# controllers/webapp_controller.go — reconciler skeleton
# config/crd/bases/             — generated CRD YAML (from controller-gen)
# config/rbac/                  — RBAC for the controller's service account

# Generate CRD YAML from your Go types (run after any type change)
make manifests

# Run locally against current kubeconfig (great for development)
make run ENABLE_WEBHOOKS=false

3. Defining the CRD Types

go — webapp_types.go 
// api/v1alpha1/webapp_types.go
package v1alpha1

import (
    metav1 "k8s.io/apimachinery/pkg/apis/meta/v1"
)

// WebAppSpec defines the desired state of WebApp
type WebAppSpec struct {
    // Image is the container image for the web application
    // +kubebuilder:validation:Required
    Image string `json:"image"`

    // Replicas is the desired number of pods. Defaults to 1.
    // +kubebuilder:validation:Minimum=0
    // +kubebuilder:validation:Maximum=100
    // +kubebuilder:default=1
    Replicas *int32 `json:"replicas,omitempty"`

    // Port is the container port to expose. Defaults to 8080.
    // +kubebuilder:default=8080
    Port int32 `json:"port,omitempty"`

    // Config holds arbitrary key-value configuration mounted as a ConfigMap.
    Config map[string]string `json:"config,omitempty"`

    // ServiceType controls how the Service is exposed. ClusterIP or LoadBalancer.
    // +kubebuilder:validation:Enum=ClusterIP;LoadBalancer;NodePort
    // +kubebuilder:default=ClusterIP
    ServiceType string `json:"serviceType,omitempty"`
}

// WebAppStatus defines the observed state of WebApp
type WebAppStatus struct {
    // Conditions store the status conditions for this WebApp.
    // +listType=map
    // +listMapKey=type
    // +patchStrategy=merge
    // +patchMergeKey=type
    Conditions []metav1.Condition `json:"conditions,omitempty"`

    // ReadyReplicas is the number of pods with Ready condition
    ReadyReplicas int32 `json:"readyReplicas,omitempty"`

    // ObservedGeneration tracks which version of the spec we last reconciled
    ObservedGeneration int64 `json:"observedGeneration,omitempty"`
}

// +kubebuilder:object:root=true
// +kubebuilder:subresource:status
// +kubebuilder:printcolumn:name="Replicas",type=integer,JSONPath=".spec.replicas"
// +kubebuilder:printcolumn:name="Ready",type=integer,JSONPath=".status.readyReplicas"
// +kubebuilder:printcolumn:name="Age",type=date,JSONPath=".metadata.creationTimestamp"
type WebApp struct {
    metav1.TypeMeta   `json:",inline"`
    metav1.ObjectMeta `json:"metadata,omitempty"`

    Spec   WebAppSpec   `json:"spec,omitempty"`
    Status WebAppStatus `json:"status,omitempty"`
}

4. Writing the Reconciler

go — webapp_controller.go 
// controllers/webapp_controller.go (key sections)
func (r *WebAppReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    log := log.FromContext(ctx)

    // 1. Fetch the WebApp instance
    webapp := &appsv1alpha1.WebApp{}
    if err := r.Get(ctx, req.NamespacedName, webapp); err != nil {
        return ctrl.Result{}, client.IgnoreNotFound(err)
        // IgnoreNotFound: if deleted, stop reconciling — no error
    }

    // 2. Reconcile ConfigMap from spec.config
    if err := r.reconcileConfigMap(ctx, webapp); err != nil {
        r.setCondition(ctx, webapp, "ConfigMapReady", metav1.ConditionFalse, "ReconcileError", err.Error())
        return ctrl.Result{}, err
    }

    // 3. Reconcile Deployment
    if err := r.reconcileDeployment(ctx, webapp); err != nil {
        r.setCondition(ctx, webapp, "DeploymentReady", metav1.ConditionFalse, "ReconcileError", err.Error())
        return ctrl.Result{}, err
    }

    // 4. Reconcile Service
    if err := r.reconcileService(ctx, webapp); err != nil {
        return ctrl.Result{}, err
    }

    // 5. Update status.readyReplicas from the Deployment status
    dep := &appsv1.Deployment{}
    if err := r.Get(ctx, req.NamespacedName, dep); err == nil {
        webapp.Status.ReadyReplicas = dep.Status.ReadyReplicas
    }
    webapp.Status.ObservedGeneration = webapp.Generation
    r.setCondition(ctx, webapp, "Available", metav1.ConditionTrue, "ReconcileSuccess", "All resources reconciled")

    if err := r.Status().Update(ctx, webapp); err != nil {
        log.Error(err, "unable to update WebApp status")
        return ctrl.Result{}, err
    }

    // Requeue after 30s to catch drift (e.g. someone manually edits the Deployment)
    return ctrl.Result{RequeueAfter: 30 * time.Second}, nil
}

func (r *WebAppReconciler) reconcileDeployment(ctx context.Context, webapp *appsv1alpha1.WebApp) error {
    desired := r.deploymentForWebApp(webapp)

    // Set WebApp as the owner of this Deployment (enables garbage collection)
    if err := ctrl.SetControllerReference(webapp, desired, r.Scheme); err != nil {
        return err
    }

    existing := &appsv1.Deployment{}
    err := r.Get(ctx, types.NamespacedName{Name: webapp.Name, Namespace: webapp.Namespace}, existing)
    if errors.IsNotFound(err) {
        return r.Create(ctx, desired)
    } else if err != nil {
        return err
    }

    // Update if spec changed (compare relevant fields, not full object)
    if !reflect.DeepEqual(existing.Spec.Template.Spec.Containers, desired.Spec.Template.Spec.Containers) ||
        *existing.Spec.Replicas != *desired.Spec.Replicas {
        existing.Spec = desired.Spec
        return r.Update(ctx, existing)
    }
    return nil
}

// Watch: reconcile when owned Deployments change, not just WebApp
func (r *WebAppReconciler) SetupWithManager(mgr ctrl.Manager) error {
    return ctrl.NewControllerManagedBy(mgr).
        For(&appsv1alpha1.WebApp{}).
        Owns(&appsv1.Deployment{}).   // re-reconcile if owned Deployment changes
        Owns(&corev1.Service{}).
        Owns(&corev1.ConfigMap{}).
        WithOptions(controller.Options{
            MaxConcurrentReconciles: 3,  // process 3 WebApps concurrently
        }).
        Complete(r)
}
Owner references are your garbage collector: Always set a controller owner reference from your operator-managed resources (Deployments, Services, ConfigMaps) back to your custom resource. When the user deletes the WebApp, Kubernetes cascading deletion will automatically clean up all owned resources. Without this, you create resource leaks that are painful to track down.

5. Status Conditions and Events

Status conditions are the standard Kubernetes way to communicate resource health. Use meta/v1.Condition with types like Available, Progressing, and Degraded. Set ObservedGeneration on every condition so users can tell if the condition reflects the current spec or a stale one.

Kubernetes Events are complementary: use them for transient messages ("Created Deployment", "Updated ConfigMap", "Waiting for pods to become ready"). Events appear in kubectl describe webapp myapp and are valuable for debugging without requiring log access. Use r.Recorder.Event(webapp, corev1.EventTypeNormal, "Created", "Deployment created") from your reconciler.

6. Leader Election for HA

In production, run multiple replicas of your operator for high availability. But you cannot run multiple reconciler instances simultaneously — they'll race. The solution is leader election: only the leader processes reconcile events; the others stand by and take over if the leader fails.

controller-runtime handles this automatically. In your main.go, enable it when creating the manager:

go — main.go (leader election) 
mgr, err := ctrl.NewManager(ctrl.GetConfigOrDie(), ctrl.Options{
    Scheme:                  scheme,
    MetricsBindAddress:      ":8080",
    HealthProbeBindAddress:  ":8081",
    LeaderElection:          true,
    LeaderElectionID:        "webapp-operator-leader",
    // Leader election uses a Lease resource in the operator's namespace
    // The controller holding the Lease is the leader
    // If it fails to renew within LeaseDuration, a standby takes over
    LeaseDuration:           &leaseDuration,   // default: 15s
    RenewDeadline:           &renewDeadline,   // default: 10s
    RetryPeriod:             &retryPeriod,     // default: 2s
})
if err != nil {
    setupLog.Error(err, "unable to start manager")
    os.Exit(1)
}

7. Unit Testing with the Fake Client

controller-runtime provides a fake client that implements the same interface as the real Kubernetes client but stores objects in memory. This lets you test your reconciler without a running cluster — fast, deterministic, no external dependencies.

go — webapp_controller_test.go 
func TestWebAppReconciler_CreatesDeployment(t *testing.T) {
    // Build a fake client pre-populated with a WebApp object
    webapp := &appsv1alpha1.WebApp{
        ObjectMeta: metav1.ObjectMeta{Name: "test-app", Namespace: "default"},
        Spec: appsv1alpha1.WebAppSpec{
            Image:    "nginx:1.25",
            Replicas: ptr(int32(2)),
            Port:     8080,
        },
    }

    fakeClient := fake.NewClientBuilder().
        WithScheme(scheme).
        WithObjects(webapp).
        WithStatusSubresource(webapp).  // enable status updates
        Build()

    reconciler := &WebAppReconciler{
        Client:   fakeClient,
        Scheme:   scheme,
        Recorder: record.NewFakeRecorder(10),
    }

    // Run the reconciler
    result, err := reconciler.Reconcile(context.Background(), ctrl.Request{
        NamespacedName: types.NamespacedName{Name: "test-app", Namespace: "default"},
    })

    require.NoError(t, err)
    assert.Equal(t, 30*time.Second, result.RequeueAfter)

    // Verify the Deployment was created
    dep := &appsv1.Deployment{}
    err = fakeClient.Get(context.Background(),
        types.NamespacedName{Name: "test-app", Namespace: "default"}, dep)
    require.NoError(t, err)
    assert.Equal(t, int32(2), *dep.Spec.Replicas)
    assert.Equal(t, "nginx:1.25", dep.Spec.Template.Spec.Containers[0].Image)

    // Verify owner reference set correctly
    assert.Len(t, dep.OwnerReferences, 1)
    assert.Equal(t, "test-app", dep.OwnerReferences[0].Name)
}

8. When NOT to Write an Operator

Operators are powerful but expensive to build and maintain. Before writing one, ask: can Helm or Kustomize + a simple Job handle this? Most stateless application deployments, even complex ones, don't need an operator. Helm handles parameterization, upgrades, and rollbacks for stateless apps well.

Write an operator when you need one of these things Helm cannot do:

  • React to observed state: "If the database replica is lagging by more than 30 seconds, promote a new primary" — requires continuous observation and action, not a one-time apply
  • Multi-step lifecycle management: Database operators (CloudNativePG, Zalando postgres-operator) handle backup scheduling, point-in-time recovery, failover — workflows that span minutes and require state
  • Self-healing without human intervention: cert-manager renews certificates before they expire; you don't run Helm to do it
  • Custom autoscaling logic: KEDA (event-driven autoscaling) reacts to queue depth, Kafka lag, or custom metrics — logic that a Deployment's HPA cannot express
Operators are software that requires maintenance: Every operator you write is production software — it needs tests, observability (metrics from the reconciler loop), alerting on reconcile errors, and upgrade procedures. I've seen teams write operators to avoid operational complexity and then create more operational complexity managing the operator itself. Only build one if the value clearly outweighs the maintenance cost.
