DevSecOps Pipeline: SAST, Container Scanning & Secrets Detection in CI/CD

Souvik· April 2026· 11 min read

Security findings discovered in production cost 30x more to remediate than findings caught at commit time — this is not a guess, it's the NIST data that's been cited since 2002 and has only become more dramatic in the cloud-native era. A leaked API key in a public GitHub repo can be found and exploited within seconds by automated scanners. A critical CVE in your base image, running in production for 6 months, can be the entry point for a supply chain attack that ends your company.

Shift-left security doesn't mean making developers into security engineers. It means putting the right automated guardrails at the right points in the pipeline so that violations are caught before they cost anyone anything. This post covers the full stack: pre-commit hooks, SAST, container scanning, secrets detection, policy-as-code, and image signing.

1. The Shift-Left Model: Where to Put Each Check

Not every security check belongs in every place. Running a full container image scan on every pre-commit hook would kill developer productivity. The correct model is tiered by speed and cost:

  • Pre-commit: Gitleaks (secrets detection), terraform fmt and terraform validate, Dockerfile linting (hadolint). Fast, local, no network. Under 5 seconds.
  • Pull Request / CI: Semgrep SAST, Trivy image scan, Conftest/OPA policy checks on manifests, dependency audit (npm audit, pip-audit, govulncheck). Can be slower — developers wait for CI, not pre-commit.
  • Post-merge / CD: Full SBOM generation, Cosign image signing, continuous scanning of deployed images (Trivy in cluster with Starboard/Trivy Operator), runtime policy enforcement (Falco, Tetragon).

2. Gitleaks: Secrets Detection at Commit Time

Gitleaks scans git history and working tree for secrets — API keys, tokens, private keys, passwords — using entropy analysis and regex rules for hundreds of known secret patterns. The most important deployment is as a pre-commit hook so secrets never enter git history at all.

yaml — .pre-commit-config.yaml 
repos:
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.21.2
    hooks:
      - id: gitleaks

  - repo: https://github.com/hadolint/hadolint
    rev: v2.12.0
    hooks:
      - id: hadolint-docker

  - repo: https://github.com/antonbabenko/pre-commit-terraform
    rev: v1.96.1
    hooks:
      - id: terraform_fmt
      - id: terraform_validate
      - id: terraform_tflint

For teams that resist pre-commit hooks, also add Gitleaks to CI as a blocking check. Scan the full PR diff: gitleaks detect --source . --log-opts "origin/main..HEAD". This catches secrets that bypass the pre-commit hook (developers working without hooks installed, IDE commits, etc.).

What to do when a secret is found in history. Removing a secret from git history with git filter-branch or BFG Repo Cleaner does NOT make it safe — it was already in the repo and may have been cloned by CI runners, mirrors, or other developers. Rotate the credential immediately. Always rotate first, then clean history. Gitleaks can also scan S3 buckets, Jira, and Confluence if you need to find historical exposure.

3. Trivy: Container Image Scanning with GitHub Actions Integration

Trivy scans container images for OS package vulnerabilities (using CVE databases), language package vulnerabilities (go.sum, package-lock.json, requirements.txt, pom.xml), misconfigurations in Dockerfile and Kubernetes manifests, and exposed secrets. It's fast, accurate, and the SARIF output integrates directly with GitHub Security tab.

yaml — .github/workflows/security.yaml 
name: Security Scanning

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 6 * * 1'   # weekly scan even without code changes

jobs:
  trivy-image-scan:
    runs-on: ubuntu-latest
    permissions:
      security-events: write
      contents: read
    steps:
      - uses: actions/checkout@v4

      - name: Build image for scanning
        run: docker build -t scan-target:${{ github.sha }} .

      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: scan-target:${{ github.sha }}
          format: sarif
          output: trivy-image.sarif
          severity: CRITICAL,HIGH
          exit-code: '1'
          ignore-unfixed: true   # don't fail on CVEs with no fix yet
          vuln-type: os,library
          trivyignores: .trivyignore  # per-repo exception list

      - name: Upload image scan results to GitHub Security
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: trivy-image.sarif
          category: trivy-image

  trivy-iac-scan:
    runs-on: ubuntu-latest
    permissions:
      security-events: write
      contents: read
    steps:
      - uses: actions/checkout@v4

      - name: Scan Kubernetes manifests and Terraform for misconfigurations
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: config
          scan-ref: .
          format: sarif
          output: trivy-config.sarif
          exit-code: '0'   # warn only for IaC misconfig — don't block

      - uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: trivy-config.sarif
          category: trivy-iac

The .trivyignore file deserves attention. Every Trivy deployment eventually accumulates CVEs that are: in a package you don't use (e.g., a web server CVE in a library you import but never expose), fixed in a newer version that's blocked by another dependency, or accepted risk documented in your security tracker. The ignore file lets you suppress these with a comment explaining why, so the signal-to-noise ratio stays high. Without it, developers start ignoring all Trivy output because it's 95% noise.

4. Semgrep: SAST with Custom Rules

Semgrep does static analysis at the AST level — it understands code structure, not just text patterns. This means it can find things like "user input passed directly to a subprocess call" across Python, Go, JavaScript, Java, and 20 other languages, without the false positives you get from grep-based approaches.

yaml — .github/workflows/semgrep.yaml (job snippet) 
semgrep:
  runs-on: ubuntu-latest
  container:
    image: semgrep/semgrep
  steps:
    - uses: actions/checkout@v4

    - name: Run Semgrep SAST
      run: |
        semgrep ci \
          --config p/security-audit \
          --config p/secrets \
          --config p/owasp-top-ten \
          --config ./semgrep-rules/ \
          --sarif --output semgrep.sarif \
          --error   # exit non-zero on findings
      env:
        SEMGREP_APP_TOKEN: ${{ secrets.SEMGREP_APP_TOKEN }}

    - uses: github/codeql-action/upload-sarif@v3
      if: always()
      with:
        sarif_file: semgrep.sarif

The ./semgrep-rules/ directory is where custom rules live. These are the most valuable part — write rules for your specific codebase anti-patterns. Example: your team has a custom logging library and you want to prevent anyone from logging request.headers['Authorization'] directly:

yaml — semgrep-rules/no-log-auth-header.yaml 
rules:
  - id: no-log-authorization-header
    message: Do not log the Authorization header — it contains bearer tokens
    severity: ERROR
    languages: [python]
    pattern-either:
      - pattern: logger.$METHOD(..., request.headers['Authorization'], ...)
      - pattern: logger.$METHOD(..., request.headers.get("Authorization"), ...)
      - pattern: print(..., request.headers['Authorization'], ...)
    metadata:
      category: security
      cwe: CWE-312
      confidence: HIGH

5. OPA/Conftest: Policy-as-Code for Kubernetes Manifests

OPA (Open Policy Agent) with Conftest lets you write policies in Rego that validate Kubernetes YAML before it reaches the cluster. This catches misconfigurations that are technically valid Kubernetes but violate your security standards: containers running as root, missing resource limits, images from untrusted registries, missing network policies.

rego — policy/kubernetes/no-root-containers.rego 
package main

deny[msg] {
  # Fail if any container runs as root (UID 0)
  container := input.spec.template.spec.containers[_]
  container.securityContext.runAsUser == 0
  msg := sprintf("Container '%s' must not run as root (runAsUser: 0)",
    [container.name])
}

deny[msg] {
  # Fail if securityContext is completely absent
  container := input.spec.template.spec.containers[_]
  not container.securityContext
  msg := sprintf("Container '%s' is missing securityContext — runAsNonRoot required",
    [container.name])
}

deny[msg] {
  # Fail if image is not from our approved registries
  container := input.spec.template.spec.containers[_]
  image := container.image
  not startswith(image, "ghcr.io/myorg/")
  not startswith(image, "123456789.dkr.ecr.us-east-1.amazonaws.com/")
  not startswith(image, "public.ecr.aws/")
  msg := sprintf("Container '%s' uses image from untrusted registry: %s",
    [container.name, image])
}

warn[msg] {
  # Warn (don't fail) if no resource limits are set
  container := input.spec.template.spec.containers[_]
  not container.resources.limits
  msg := sprintf("Container '%s' has no resource limits set", [container.name])
}
yaml — GitHub Actions: Conftest job 
conftest:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4

    - name: Install conftest
      run: |
        curl -Lo conftest.tar.gz https://github.com/open-policy-agent/conftest/releases/download/v0.56.0/conftest_0.56.0_Linux_x86_64.tar.gz
        tar xzf conftest.tar.gz conftest
        chmod +x conftest && sudo mv conftest /usr/local/bin/

    - name: Validate Kubernetes manifests against policies
      run: |
        conftest test \
          --policy policy/kubernetes/ \
          --namespace main \
          k8s/
Start with warnings, then escalate to denials. When you first introduce OPA policies into a pipeline, the number of existing violations is usually daunting. Start with warn rules for everything. Give teams 30-60 days to fix their existing manifests. Then flip the rules to deny. This avoids the "security team blocked my deployment" backlash that kills DevSecOps adoption programs.

6. Cosign: Image Signing and Verification

Cosign (from the Sigstore project) signs container images using keyless signing backed by OIDC. In a GitHub Actions workflow, the GHA OIDC token is used as the identity — no key management required. The signature is stored in the OCI registry alongside the image. At admission time, a Kubernetes admission controller (Kyverno or OPA Gatekeeper) verifies the signature before allowing the pod to start.

yaml — GitHub Actions: sign image after push 
sign-image:
  needs: [build-scan-push]
  runs-on: ubuntu-latest
  permissions:
    packages: write
    id-token: write   # required for keyless signing
  steps:
    - name: Install Cosign
      uses: sigstore/cosign-installer@v3

    - name: Sign the container image (keyless)
      run: |
        cosign sign --yes \
          ghcr.io/myorg/api@${{ needs.build-scan-push.outputs.image-digest }}
      # Signs using the GitHub Actions OIDC identity
      # Signature stored in ghcr.io/myorg/api:sha256-abc123.sig
      # Transparency log entry created in Rekor for public auditability

    - name: Attach SBOM to image
      run: |
        syft ghcr.io/myorg/api@${{ needs.build-scan-push.outputs.image-digest }} \
          -o spdx-json > sbom.spdx.json
        cosign attach sbom \
          --sbom sbom.spdx.json \
          ghcr.io/myorg/api@${{ needs.build-scan-push.outputs.image-digest }}
        cosign sign --yes \
          --attachment sbom \
          ghcr.io/myorg/api@${{ needs.build-scan-push.outputs.image-digest }}

The Kyverno ClusterPolicy that enforces signature verification at admission time:

yaml — kyverno ClusterPolicy: require signed images 
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-signed-images
spec:
  validationFailureAction: Enforce
  background: false
  rules:
    - name: check-image-signature
      match:
        any:
          - resources:
              kinds: [Pod]
              namespaces:
                - api-prod
                - payments-prod
      verifyImages:
        - imageReferences: ["ghcr.io/myorg/*"]
          attestors:
            - entries:
                - keyless:
                    subject: "https://github.com/myorg/*/.github/workflows/*.yaml@refs/heads/main"
                    issuer: "https://token.actions.githubusercontent.com"

7. When to Fail vs When to Warn

The failure policy matters as much as the detection capability. A pipeline that blocks on every finding gets bypassed. A pipeline that only warns on everything is useless. Here's the policy I apply in production:

  • Hard fail (block merge): Gitleaks secret detection, Trivy CRITICAL severity CVEs with a fix available, Cosign signature verification failure, Conftest policy denial rules (privileged containers, root user, disallowed registries), Semgrep HIGH/CRITICAL findings.
  • Warn (comment on PR, do not block): Trivy HIGH severity CVEs without a fix, Trivy MEDIUM, Semgrep MEDIUM and informational findings, Conftest warn rules (missing resource limits, missing readinessProbe), dependency upgrade suggestions.
  • Never run in pre-commit: Container image scanning (too slow), full Semgrep ruleset (too slow). Pre-commit should complete in under 10 seconds or developers disable it.
Security debt tracking. Every .trivyignore exception, every Semgrep suppression (# nosemgrep), and every accepted Gitleaks false positive should be tracked in your issue tracker with a due date for resolution. A quarterly security debt review that removes stale suppressions is worth scheduling. Without this, the ignore files grow indefinitely and the scanner becomes noise.

8. SBOM Generation with Syft

An SBOM (Software Bill of Materials) is a machine-readable inventory of every library, package, and component in your container image. It's required for SLSA compliance, increasingly required by enterprise customers, and practically useful for quickly answering "are we affected by CVE-XXXX-YYYY?" when the next Log4Shell-type vulnerability hits.

Generate it with Syft as part of your post-push CI step and attach it to the image with Cosign. When a new CVE is announced, you can query all SBOMs in your registry to find affected services in minutes rather than days:

bash — query SBOMs for CVE exposure 
# Download SBOM for a specific image
cosign download sbom ghcr.io/myorg/api:sha-abc123 > api.sbom.json

# Query with grype to check for a specific CVE across the SBOM
grype sbom:api.sbom.json --only-fixed --by-cve CVE-2021-44228

# Or use syft directly against a running image
syft ghcr.io/myorg/api:sha-abc123 -o table | grep -i log4j

# Batch: check all production images for a CVE
for service in api payments worker cron; do
  echo "=== $service ==="
  cosign download sbom ghcr.io/myorg/${service}:latest 2>/dev/null \
    | grype --quiet --by-cve CVE-2021-44228
done

Summary

A mature DevSecOps pipeline is not a single tool — it's a layered system where each control has a clear purpose and a clear failure policy. Gitleaks and pre-commit hooks prevent secrets from entering history. Trivy and Semgrep catch vulnerabilities and code issues at PR time. Conftest enforces Kubernetes security standards before manifests reach the cluster. Cosign and Kyverno ensure only CI-built, signed images run in production. And Syft gives you the SBOM inventory that makes CVE response a query, not a crisis.

The common failure mode is implementing all of these as "warn only" because nobody wants to block deployments. Resist that. Start with a small, high-confidence set of hard failures — leaked secrets and exploitable CRITICAL CVEs — and expand from there as the team builds trust in the tooling.
