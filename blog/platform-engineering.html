Platform Engineering vs DevOps: Building an Internal Developer Platform

Souvik· January 2026· 12 min read

"DevOps" promised to eliminate the wall between development and operations. What it actually produced, in many organizations, was a wall between developers and the growing complexity of cloud infrastructure — wrapped in the expectation that every developer should understand Kubernetes, Terraform, IAM policies, service meshes, and observability pipelines on top of building product features. The cognitive load is real, and it kills velocity.

Platform engineering is the answer to that problem. Not a replacement for DevOps culture — DevOps is still the right mental model for breaking down silos and owning what you build in production — but a product discipline applied to internal tooling. The platform team builds and operates the Internal Developer Platform (IDP) so that product engineers can deploy, observe, and manage their services without becoming infrastructure specialists.

1. The Cognitive Load Problem

Team Topologies (Skelton & Pais) describes three types of cognitive load: intrinsic (understanding the problem domain), extraneous (accidental complexity from tools), and germane (building capability). Platform engineering's job is to minimize extraneous cognitive load on stream-aligned teams so they can focus on intrinsic and germane load — i.e., building good software.

In practice, this looks like: a developer should not need to understand Crossplane, Kubernetes RBAC, Terraform state backends, or Helm chart templating to deploy a new microservice to production. They should fill out a form (or run a CLI command), get back a working deployment with CI/CD, monitoring, and a service catalog entry, and ship features. The platform team owns the complexity; developers consume a product.

The key shift from traditional DevOps thinking: the platform team treats developers as their customers. That means user research (how long does it take to deploy a new service today?), product roadmaps, SLAs for the platform itself, and user satisfaction metrics. Not just "we wrote the Helm chart, use it."

2. The Golden Path Concept

A golden path (or paved road) is an opinionated, well-supported path through your infrastructure for the common case. It bundles all the best practices: security scanning in CI, automatic mTLS via service mesh, centralized logging and metrics, RBAC by convention, automated certificate management, and a standard Dockerfile that passes your security policy.

Developers who follow the golden path get all of these for free. Developers who need to deviate — because they have unusual requirements, legacy systems, or specialized technology needs — can, but they accept ownership of the complexity they're introducing. The platform team doesn't block deviation; it makes the golden path so good that most teams never need to deviate.

The golden path must actually be golden: If the "standard" Helm chart has 200 values, requires reading three runbooks to get right, and takes 4 hours to debug the first time, it's not a golden path — it's a golden obstacle course. Invest in making the happy path genuinely fast and pleasant. Measure onboarding time for new services. If it takes more than 30 minutes for a standard service, your platform has a UX problem.

3. Backstage: The Developer Portal Layer

Backstage (open-sourced by Spotify, now a CNCF project) provides the user-facing layer of an IDP. Its core components are:

  • Service Catalog: A searchable inventory of all services, libraries, APIs, and data pipelines in your organization. Each entry is backed by a catalog-info.yaml file in the service's Git repo.
  • TechDocs: Docs-as-code — Markdown docs in the service's repo, rendered as a documentation portal in Backstage. No separate Confluence page needed.
  • Scaffolder Templates: Wizard-driven project creation that can scaffold a repo, create a PR, provision cloud resources, and register the new service in the catalog — in one workflow.
  • Plugins: First-class integrations with Kubernetes (showing pod health in the catalog), PagerDuty (showing active incidents per service), SonarQube, Lighthouse, and 100+ community plugins.
yaml — catalog-info.yaml 
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: payment-api
  description: "Handles payment processing via Stripe and Adyen"
  annotations:
    backstage.io/techdocs-ref: dir:.
    # Kubernetes plugin: shows pods, deployments, health in Backstage
    backstage.io/kubernetes-id: payment-api
    backstage.io/kubernetes-namespace: production
    # PagerDuty plugin: link to on-call schedule
    pagerduty.com/service-id: P1ABC23
    # SonarQube code quality gate
    sonarqube.org/project-key: payment-api
    # ArgoCD plugin: link to deployment history
    argocd/app-name: payment-api-prod
  tags:
  - payments
  - critical
  - tier-1
  links:
  - url: "https://grafana.internal/d/payment-api"
    title: Grafana Dashboard
  - url: "https://logs.internal/app/payment-api"
    title: Kibana Logs
spec:
  type: service
  lifecycle: production
  owner: team-payments
  system: checkout-platform
  dependsOn:
  - component:postgres-payment
  - component:stripe-connector
  providesApis:
  - payment-api-v2

4. Crossplane: Self-Service Infrastructure

Backstage handles the portal and catalog layer. For self-service infrastructure provisioning, Crossplane is the tool that connects Kubernetes GitOps workflows to cloud provider APIs. Developers create a custom resource (e.g., PostgresDatabase); Crossplane's composition engine translates it into the appropriate provider-specific resources (RDS instance, subnet group, parameter group, IAM role, Secrets Manager entry) and reconciles them continuously.

The two key Crossplane primitives are CompositeResourceDefinitions (XRDs) — the API the developer sees — and Compositions — the template that maps the XRD to real cloud resources.

yaml — crossplane-composition.yaml 
# XRD: defines the self-service API your developers use
apiVersion: apiextensions.crossplane.io/v1
kind: CompositeResourceDefinition
metadata:
  name: xpostgresdatabases.platform.example.com
spec:
  group: platform.example.com
  names:
    kind: XPostgresDatabase
    plural: xpostgresdatabases
  claimNames:
    kind: PostgresDatabase       # namespace-scoped claim — developers use this
    plural: postgresdatabases
  versions:
  - name: v1alpha1
    served: true
    referenceable: true
    schema:
      openAPIV3Schema:
        type: object
        properties:
          spec:
            type: object
            properties:
              parameters:
                type: object
                properties:
                  size:
                    type: string
                    enum: [small, medium, large]  # t3.micro, t3.small, t3.medium
                    description: "Instance size tier"
                  version:
                    type: string
                    enum: ["14", "15", "16"]
                  multiAZ:
                    type: boolean
                    default: false
                required: [size, version]

---
# Composition: platform team's implementation — developers never see this
apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: postgres-aws-rds
spec:
  compositeTypeRef:
    apiVersion: platform.example.com/v1alpha1
    kind: XPostgresDatabase
  resources:
  - name: rds-instance
    base:
      apiVersion: rds.aws.upbound.io/v1beta1
      kind: Instance
      spec:
        forProvider:
          region: us-east-1
          engine: postgres
          storageEncrypted: true             # security policy enforced by platform
          backupRetentionPeriod: 7
          deletionProtection: true
          dbSubnetGroupNameSelector:
            matchLabels:
              env: production
    patches:
      # Map developer's "small/medium/large" to actual instance types
    - type: CombineFromComposite
      combine:
        variables:
        - fromFieldPath: spec.parameters.size
        strategy: string
        string:
          fmt: "%s"
      toFieldPath: spec.forProvider.instanceClass
      transforms:
      - type: map
        map:
          small: db.t3.micro
          medium: db.t3.small
          large: db.t3.medium
    - fromFieldPath: spec.parameters.version
      toFieldPath: spec.forProvider.engineVersion
    - fromFieldPath: spec.parameters.multiAZ
      toFieldPath: spec.forProvider.multiAz

With this in place, a developer provisions a database by applying a simple YAML claim to their namespace. The platform team has enforced storage encryption, backup retention, subnet placement, and deletion protection — invisible to the developer, non-negotiable by policy.

5. Backstage Scaffolder: The "New Service" Golden Path

A Backstage Scaffolder template automates the entire "create a new service" workflow. When a developer fills out the form, Backstage can: clone a template repo, replace placeholders, create a new GitHub repo, push the scaffolded code, create a PR, provision a staging namespace via Crossplane, register the service in the Backstage catalog, and create the Grafana dashboard — all from a single workflow defined in YAML and executed by the Scaffolder backend.

This is the single highest-leverage investment a platform team can make. The first service a new team creates sets the tone for everything that follows. If it takes 3 minutes using Backstage Scaffolder vs 3 days of back-and-forth with the platform team, you've earned the trust of every developer in the organization.

6. Tekton for Internal Pipelines

For CI/CD pipelines that developers run but the platform team owns, Tekton is the Kubernetes-native answer. Unlike GitHub Actions (which run on GitHub's infrastructure) or Jenkins (which requires managing its own cluster), Tekton pipelines are Kubernetes resources — they run in your cluster, access your secrets natively, and are defined in YAML that can be versioned and policy-checked like any other resource.

The platform team defines reusable Tasks (e.g., "build Docker image with Kaniko", "run Trivy security scan", "deploy to staging via ArgoCD sync") and composes them into Pipelines. Teams consume these pre-built pipelines via PipelineRun objects, parameterized with their service's image name and repo. They get a standardized, compliant pipeline without writing a single pipeline definition.

7. Measuring IDP Success with DORA Metrics

The point of a platform is to make developers faster and more reliable. DORA metrics (from the DevOps Research and Assessment program) are the industry-standard way to measure this:

  • Deployment Frequency: How often does a team deploy to production? Elite performers deploy on-demand (multiple times per day). If your IDP reduces deployment friction, this goes up.
  • Lead Time for Changes: Time from code commit to production. A good IDP + golden path should target elite performance: less than one hour for standard changes.
  • Change Failure Rate: Percentage of deployments causing an incident. Your golden path's built-in testing and canary deployment capabilities directly improve this.
  • Mean Time to Restore (MTTR): How quickly can teams recover from incidents? Backstage's integration with PagerDuty and the service catalog makes it faster to identify the owning team and relevant runbooks during incidents.

Baseline these metrics before launching the IDP. Measure them quarterly. If Deployment Frequency isn't going up and Lead Time isn't going down 6 months after launch, your IDP has a product problem — talk to developers and find out what's blocking them.

promql — dora-lead-time-query.txt 
# Lead Time: time from first commit in PR to production deployment
# Requires your CI/CD system to emit events with commit SHA and deploy timestamp

# If using Tekton + custom metrics:
histogram_quantile(0.50,
  sum(rate(deployment_lead_time_seconds_bucket{env="production"}[7d])) by (le, team)
)
# P50 lead time per team, 7-day window

# Deployment frequency: deploys per day per team
sum(increase(deployments_total{env="production"}[24h])) by (team)

# Change failure rate: deployments that triggered a rollback or incident
sum(increase(deployment_failures_total{env="production"}[7d])) by (team)
/
sum(increase(deployments_total{env="production"}[7d])) by (team)

8. Team Topologies: Platform Team as a Product Team

Team Topologies prescribes that the platform team should operate as a product team with a long-lived mandate, not a project team that hands off and disbands. This means a dedicated roadmap, quarterly OKRs aligned to developer productivity, user research interviews with stream-aligned teams, and a proper backlog prioritized by developer pain.

The platform team's customers are internal. That's a different muscle than building external products, but the same discipline applies: talk to users, measure outcomes, prioritize ruthlessly, and don't build things no one asked for.

9. Common IDP Pitfalls

Building the IDP in secret and launching with a big bang: Platform teams sometimes spend 12 months building the perfect IDP before showing it to developers. By then, teams have built workarounds, accumulated tribal knowledge, and have zero incentive to switch. Ship an MVP to 2-3 pilot teams within 8 weeks, iterate from there.

Treating the IDP as a technology project, not a product: The platform is not done when the Backstage instance is running. It's done when developers say "this makes my job better." Until then, you're iterating.

Over-engineering the abstractions: Crossplane compositions that try to abstract every possible configuration option end up being as complex as the underlying Terraform they replaced. Start with narrow, opinionated abstractions that cover 80% of use cases. Add flexibility incrementally based on actual developer requests, not anticipated ones.

Avoid the "second system effect" trap: The first IDP tool a platform team builds tends to be narrowly useful. The second one is overengineered to be a universal abstraction layer. Fight the urge to build a general-purpose infrastructure API. Build specific, well-scoped primitives: a Postgres claim, a Redis claim, a standard Deployment template. Generality is the enemy of usability.
P75 time-to-production as your North Star: Track the 75th percentile (not the mean) of time-to-first-production-deployment for a new service. The mean is skewed by the fastest teams who know the platform well. P75 captures the experience of teams encountering your IDP for the first time. If your P75 is measured in days, you have usability problems. Target sub-4-hours for a new standard service.
