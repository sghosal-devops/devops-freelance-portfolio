Terraform at Scale: State Management, Modules & Multi-Account AWS Architecture

Souvik· June 2026· 10 min read

Running Terraform for a single environment on a single AWS account is a solved problem. Running it across 20+ accounts, 6 environments, and 4 teams — without state collisions, blast-radius disasters, or module version chaos — is a completely different engineering challenge. I've done this migration twice now, and each time there were the same set of footguns. This post is the distillation of those hard lessons.

We'll cover remote state with proper locking, the workspace trap, module versioning discipline, PR-gated workflows with Atlantis, and how to actually vend AWS accounts at scale with Control Tower.

1. Remote State with S3 + DynamoDB — Getting It Right

The default local backend is fine for learning. In production, it means one engineer holds the state file in their .terraform directory and nobody else can run plan safely. Switch to S3 + DynamoDB on day one.

The S3 bucket configuration matters more than most guides admit. You need versioning (so you can roll back a corrupted state), encryption at rest (KMS, not AES256 — you want audit trails on key usage), and block all public access. The DynamoDB table only needs a single attribute: LockID as the partition key, String type. That's it — Terraform manages the rest.

hcl — backend.tf 
terraform {
  backend "s3" {
    bucket         = "myorg-terraform-state-prod"
    key            = "services/api-gateway/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    kms_key_id     = "arn:aws:kms:us-east-1:123456789012:key/mrk-abc123"
    dynamodb_table = "terraform-state-locks"

    # Assume this role when accessing state — cross-account pattern
    role_arn       = "arn:aws:iam::111111111111:role/TerraformStateRole"
    session_name   = "TerraformCICD"
  }
}

Notice the role_arn. In a multi-account setup, your state bucket lives in a dedicated tooling account. Every other account assumes a role into that tooling account to read/write state. This gives you centralised audit logs via CloudTrail on the state bucket, and you can restrict which OIDC principals (GitHub Actions, Atlantis) can assume which roles.

Bootstrap the state infrastructure itself with a small, hand-managed Terraform root — or just do it once with the AWS CLI and never touch it again. The state for your state bucket should not be stored in itself.

hcl — state-bootstrap/main.tf 
resource "aws_s3_bucket" "state" {
  bucket = "myorg-terraform-state-prod"
}

resource "aws_s3_bucket_versioning" "state" {
  bucket = aws_s3_bucket.state.id
  versioning_configuration {
    status = "Enabled"
  }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "state" {
  bucket = aws_s3_bucket.state.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm     = "aws:kms"
      kms_master_key_id = aws_kms_key.state.arn
    }
  }
}

resource "aws_dynamodb_table" "locks" {
  name         = "terraform-state-locks"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"

  attribute {
    name = "LockID"
    type = "S"
  }

  point_in_time_recovery {
    enabled = true
  }
}
State file = crown jewels. A corrupted or accidentally-deleted state file means Terraform no longer knows what it manages. Enable S3 Object Lock (Compliance mode) on the state bucket and keep a 90-day PITR window on the DynamoDB table. If you ever need to manually edit state, use terraform state mv and terraform state rm — never edit the JSON directly unless you're desperate and have a backup.

2. Why You Should NOT Use Terraform Workspaces for Environments

Workspaces sound perfect for environment management: one codebase, switch workspace, different state. I've seen this pattern cause production incidents at two companies. Here's why it's a trap.

The core problem: workspaces share the same configuration. If you run terraform workspace select prod && terraform apply with a dev variable file by mistake, you've just applied dev config to production. The only safeguard is human discipline — and that's not a safeguard.

Second problem: workspace names are not enforced. Anyone can create a terraform workspace new prd (typo), apply there, and now you have two states claiming to manage overlapping resources. The collision is silent until something breaks.

The correct pattern is directory-based separation with shared modules:

bash — recommended repo structure 
# Instead of workspaces, use directories per environment
infrastructure/
├── modules/
│   ├── eks-cluster/          # versioned, reusable modules
│   ├── rds-postgres/
│   └── vpc-standard/
├── environments/
│   ├── dev/
│   │   ├── backend.tf        # points to dev state key
│   │   ├── main.tf
│   │   └── terraform.tfvars  # dev-specific values
│   ├── staging/
│   │   ├── backend.tf        # points to staging state key
│   │   ├── main.tf
│   │   └── terraform.tfvars
│   └── prod/
│       ├── backend.tf        # points to prod state key
│       ├── main.tf
│       └── terraform.tfvars
└── account-vending/          # Control Tower customizations

Each environment directory has its own backend config pointing to a different S3 key and different DynamoDB lock. You cannot accidentally cross environments because the state is structurally separated. CI/CD only has permission to apply to the environment matching the branch or PR label.

3. Module Versioning with a Private Registry

Public modules from the Terraform Registry are fine for prototyping. In production you want a private registry so you control the module surface area, force internal security standards, and don't get surprised by upstream breaking changes.

The simplest private registry is a Git tag convention on a monorepo or module repo. Terraform supports the git::https:// source with ?ref= for tags. For a proper registry UI and semantic versioning enforcement, use Terraform Cloud/Enterprise or Spacelift. For a self-hosted option, Gitea with the module registry plugin works well.

hcl — environments/prod/main.tf 
module "vpc" {
  source  = "git::https://github.com/myorg/terraform-modules.git//modules/vpc-standard?ref=v3.2.1"

  cidr_block           = "10.10.0.0/16"
  availability_zones   = ["us-east-1a", "us-east-1b", "us-east-1c"]
  enable_nat_gateway   = true
  single_nat_gateway   = false  # prod: one per AZ for HA
}

module "eks" {
  source  = "git::https://github.com/myorg/terraform-modules.git//modules/eks-cluster?ref=v5.1.0"

  cluster_name    = "prod-us-east-1"
  cluster_version = "1.30"
  vpc_id          = module.vpc.vpc_id
  subnet_ids      = module.vpc.private_subnet_ids

  node_groups = {
    general = {
      instance_types = ["m6i.xlarge", "m6a.xlarge"]
      min_size       = 3
      max_size       = 20
      desired_size   = 6
    }
  }
}

Pin modules to exact tags in production — never to a branch. Create a CODEOWNERS file for the modules directory so any version bump requires review from the platform team. Run terraform init -upgrade only intentionally, not as part of routine CI.

Module design rule: A module should manage one logical unit of infrastructure. If your module has more than ~5 resources, ask whether it should be split. The worst modules I've seen try to vend an entire application stack — VPC, EKS, RDS, Route53, ACM — in one module. When something breaks during plan, you have no idea where to look.

4. Atlantis: PR-Based Plan and Apply

Running terraform apply from a developer's laptop is how you get state drift, unreviewed changes, and no audit trail. Atlantis solves this by hooking into your VCS: it runs terraform plan on every PR and posts the output as a PR comment. Apply only happens via a comment on a merged PR, in a locked workflow.

yaml — atlantis.yaml (repo root) 
version: 3
automerge: false
delete_source_branch_on_merge: true

projects:
  - name: prod-vpc
    dir: environments/prod
    workspace: default
    terraform_version: v1.8.5
    autoplan:
      when_modified:
        - "**/*.tf"
        - "**/*.tfvars"
        - "../../modules/**/*.tf"  # replan if shared module changes
      enabled: true
    apply_requirements:
      - mergeable      # PR must be approved and all checks green
      - approved       # at least one approval required
    workflow: custom-prod

workflows:
  custom-prod:
    plan:
      steps:
        - env:
            name: AWS_ROLE_ARN
            value: arn:aws:iam::999999999999:role/AtlantisProdRole
        - init
        - plan:
            extra_args: ["-var-file=terraform.tfvars"]
    apply:
      steps:
        - apply

Atlantis should run in its own namespace in your EKS cluster with an IRSA role that has tightly scoped permissions. Never give Atlantis AdministratorAccess — define an IAM policy with only what your Terraform code actually touches, and audit it quarterly. Use the --repo-allowlist flag to restrict which repos can trigger it.

5. Provider Aliasing for Multi-Account Operations

Sometimes a single Terraform root needs to create resources in multiple AWS accounts — for example, creating a Route53 record in the shared services account while deploying an ALB in the workload account. Provider aliases handle this cleanly.

hcl — providers.tf 
provider "aws" {
  region = "us-east-1"
  # Default provider — workload account
  assume_role {
    role_arn = "arn:aws:iam::${var.workload_account_id}:role/TerraformRole"
  }
}

provider "aws" {
  alias  = "dns"
  region = "us-east-1"
  # Alias provider — shared DNS account
  assume_role {
    role_arn = "arn:aws:iam::${var.dns_account_id}:role/TerraformDNSRole"
  }
}

# Use in resource:
resource "aws_route53_record" "api" {
  provider = aws.dns   # targets the DNS account
  zone_id  = var.hosted_zone_id
  name     = "api.myorg.com"
  type     = "A"
  alias {
    name                   = aws_lb.api.dns_name
    zone_id                = aws_lb.api.zone_id
    evaluate_target_health = true
  }
}

6. Handling Sensitive Outputs

Marking an output as sensitive = true prevents it from appearing in plan/apply output, but it does NOT encrypt it in state. Anyone with read access to the state bucket can see it. The actual solution is to not put secrets into Terraform state at all.

Generate secrets outside Terraform (with AWS Secrets Manager, or a random_password resource that writes directly to Secrets Manager), and reference them by ARN rather than by value. For database passwords: create the password in Secrets Manager, pass the ARN to your RDS module, RDS reads it. Terraform never holds the plaintext.

hcl — secrets pattern 
resource "random_password" "db" {
  length  = 32
  special = true
}

resource "aws_secretsmanager_secret" "db_password" {
  name                    = "/prod/rds/api-db/password"
  recovery_window_in_days = 7
}

resource "aws_secretsmanager_secret_version" "db_password" {
  secret_id     = aws_secretsmanager_secret.db_password.id
  secret_string = random_password.db.result  # stored in state — but encrypted at rest by KMS
}

resource "aws_db_instance" "api" {
  manage_master_user_password   = true  # RDS-native rotation — best option
  master_user_secret_kms_key_id = aws_kms_key.rds.arn
  # password never touches Terraform state at all
}

7. Account Vending with AWS Control Tower

At scale, manually creating AWS accounts is a bottleneck. Control Tower's Account Factory — combined with the Account Factory for Terraform (AFT) open-source module — lets you vend new AWS accounts by merging a PR. The workflow is: engineer creates a PR adding an account definition YAML, Atlantis picks it up, AFT provisions the account via the Control Tower API, applies baseline guardrails (SCPs, CloudTrail, Config), and registers it in the organization.

hcl — aft-account-request/accounts/payments-prod.tf 
module "payments_prod" {
  source = "./modules/aft-account-request"

  control_tower_parameters = {
    AccountEmail              = "aws-payments-prod@myorg.com"
    AccountName               = "payments-prod"
    ManagedOrganizationalUnit = "Production (ou-xxxx-yyyyyyy)"
    SSOUserEmail              = "platform-team@myorg.com"
    SSOUserFirstName          = "Platform"
    SSOUserLastName           = "Team"
  }

  account_tags = {
    Environment  = "Production"
    CostCenter   = "payments"
    DataClass    = "PCI-DSS"
    ManagedBy    = "AFT"
  }

  change_management_parameters = {
    change_requested_by = "platform-team"
    change_reason       = "New payments production account"
  }

  account_customizations_name = "pci-workload"  # applies PCI baseline SCP + Config rules
}

Each account gets a baseline customization that applies your organization's standard SCPs (deny root usage, deny disabling CloudTrail, require MFA for console), enables Config, enables Security Hub, and sets up centralized logging to the log archive account. All of this runs automatically — the requesting team just gets a new account that's already compliant.

Drift detection: Run terraform plan in CI on a schedule (daily or weekly) across all environment roots, even when there are no code changes. AWS console changes, manual remediation, and upstream provider updates all cause drift. Catching it early — before the next apply — prevents configuration surprises. Atlantis supports drift detection mode natively since version 0.27.

Summary

Terraform at scale is not about clever HCL — it's about structure, access control, and discipline. Use S3+DynamoDB for state with KMS encryption. Separate environments by directory, not workspace. Pin module versions. Gate applies behind PR review with Atlantis. Use provider aliases for multi-account operations. Keep secrets out of state by using AWS Secrets Manager natively. And vend new accounts through AFT rather than manually clicking through the console.

The teams I've seen succeed at this are the ones who treat their Terraform repos like production software: code review, versioning, automated testing (Terratest or terraform validate + OPA), and runbooks for common break-glass scenarios.
