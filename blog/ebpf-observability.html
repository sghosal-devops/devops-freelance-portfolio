eBPF for DevOps Engineers: Kernel-Level Observability Without Code Changes

Souvik· May 2026· 11 min read

Every time I've joined a new production incident, the first 20 minutes are the same: where is the problem? Application metrics say latency spiked. The app team says their code hasn't changed. Infrastructure team says CPU and memory look fine. Network team is already drafting their "not us" reply. Meanwhile, something at the kernel level — a scheduler contention, a TCP retransmit storm, a syscall being blocked — is silently wrecking your p99 latency, and your existing observability stack can't see it.

eBPF is the technology that ends that conversation. This post is a practical introduction for DevOps engineers who want to actually use it, not just know it exists.

1. What eBPF Actually Is

eBPF (extended Berkeley Packet Filter) is a Linux kernel subsystem that lets you run sandboxed programs inside the kernel — without modifying kernel source code and without loading kernel modules. The kernel verifies the program before loading it: it checks for infinite loops, out-of-bounds memory access, and illegal operations. If the verifier approves it, the program is JIT-compiled and runs at near-native speed in kernel context.

The three core primitives you need to understand:

  • BPF Programs: Small C-like functions attached to kernel hook points (tracepoints, kprobes, XDP hooks, socket hooks). They run when the hook fires — on every syscall, every packet received, every scheduler event.
  • BPF Maps: Key-value data structures shared between the eBPF program (running in kernel space) and a userspace process. This is how data flows out — your eBPF program writes per-PID latency data to a map, your userspace collector reads it and exports to Prometheus.
  • BPF Hooks: The attachment points. Kprobes attach to any kernel function. Uprobes attach to userspace functions. Tracepoints are stable, predefined kernel instrumentation points. XDP hooks run at the network driver level, before the kernel networking stack even sees the packet.
Kernel version matters. Most production-useful eBPF features require Linux 4.18+. Full BTF (BPF Type Format) support — which enables CO-RE, allowing eBPF programs to run across kernel versions without recompilation — requires 5.4+. EKS defaults to Amazon Linux 2 with kernel 5.10 as of 2025. GKE uses 5.15+. Check your kernel version: uname -r.

2. bpftrace: Live Kernel Tracing with One-Liners

bpftrace is the awk of eBPF — a high-level tracing language that compiles to eBPF programs. You write a one-liner, it attaches to a kernel hook, collects data, and prints results. No C, no compilation pipeline, no restart required.

Install it on any modern Linux: apt install bpftrace or yum install bpftrace. Then:

bash — bpftrace one-liners 
# Trace all open() syscalls and show which files are being opened
bpftrace -e 'tracepoint:syscalls:sys_enter_openat { printf("%s %s\n", comm, str(args->filename)); }'

# Show disk I/O latency histogram per process (in microseconds)
bpftrace -e 'kretprobe:blk_account_io_start { @[comm] = hist(nsecs); }'

# Count TCP retransmits per destination IP — find flapping connections
bpftrace -e 'kprobe:tcp_retransmit_skb { @[ntop(AF_INET, ((struct sock *)arg0)->__sk_common.skc_daddr)] = count(); }'

# Trace which processes are calling execve (process creation) — good for detecting unexpected spawns
bpftrace -e 'tracepoint:syscalls:sys_enter_execve { printf("%-6d %-16s %s\n", pid, comm, str(args->filename)); }'

# Measure read() syscall latency histogram for a specific PID
bpftrace -e 'tracepoint:syscalls:sys_enter_read /pid == 12345/ { @start[tid] = nsecs; }
tracepoint:syscalls:sys_exit_read /pid == 12345 && @start[tid]/ {
  @latency_us = hist((nsecs - @start[tid]) / 1000);
  delete(@start[tid]);
}'

# Track off-CPU time — why is my process sleeping and for how long?
bpftrace -e 'tracepoint:sched:sched_switch {
  if (args->prev_state == TASK_RUNNING && args->prev_comm == "gunicorn") {
    @offcpu[args->next_comm] = count();
  }
}'

That last one is a production war story. A Python API was showing mysterious 50ms latency spikes. CPU was 20%, memory fine, no errors. The off-CPU bpftrace showed the process was sleeping waiting on futex calls — a locking contention issue in the ORM's connection pool. Took 10 minutes to diagnose with eBPF; the previous approach had been adding more logging and waiting for the next spike.

3. Cilium: eBPF-Powered Kubernetes Networking

Cilium replaces kube-proxy and the iptables-based networking stack with eBPF programs at the XDP and TC (Traffic Control) hook points. This gives you several concrete wins: lower latency (fewer hops through the networking stack), better throughput (XDP processes packets before kernel buffers), and native network policy enforcement at the eBPF layer.

The practical DevOps story with Cilium is Hubble — the observability layer built on top of Cilium's eBPF programs. Hubble gives you L7 visibility into every flow between pods with zero application changes.

bash — Hubble CLI commands 
# Install Cilium + Hubble via Helm
helm install cilium cilium/cilium --version 1.16.0 \
  --namespace kube-system \
  --set hubble.relay.enabled=true \
  --set hubble.ui.enabled=true \
  --set kubeProxyReplacement=true \
  --set k8sServiceHost=10.0.0.1 \
  --set k8sServicePort=443

# Forward Hubble relay port
kubectl port-forward -n kube-system svc/hubble-relay 4245:80

# Watch all HTTP traffic from the api namespace in real time
hubble observe --from-namespace api-prod --protocol http

# Find all flows that were dropped by network policy
hubble observe --verdict DROPPED --last 500

# Show TCP connections between specific pods (L4 flow)
hubble observe \
  --from-pod api-prod/api-7d4b8f9 \
  --to-pod payments-prod/payments-6c9d2a1 \
  --protocol tcp

# Get per-pod HTTP error rates (L7 metrics)
hubble observe --type l7 --protocol http \
  --http-status-code 5[0-9][0-9] --last 1000
Cilium Network Policy vs Kubernetes NetworkPolicy: Cilium supports standard Kubernetes NetworkPolicy objects, but also has its own CiliumNetworkPolicy CRD that adds L7 filtering. You can write a policy that allows HTTP GET to /api/v1/health but denies POST — iptables-based solutions can't do this because they don't understand HTTP. For zero-trust microservices architectures, this is enormously powerful.

4. Pixie: Full-Body Kubernetes Observability

Pixie is an open-source Kubernetes observability platform from New Relic that uses eBPF to automatically capture request/response bodies, latency, errors, and resource usage — no instrumentation needed. It uses uprobes to intercept TLS at the userspace crypto library level (Go's crypto/tls, OpenSSL) so it can decrypt and inspect HTTPS traffic without needing the certificate private key.

Pixie uses PxL (Pixie Language), a Python-like query language for its data. You write scripts that run against the live cluster data:

python — PxL: HTTP latency by service 
# PxL script: p50/p90/p99 HTTP latency per Kubernetes service
import px

df = px.DataFrame('http_events', start_time='-5m')

# Annotate with pod and service info from Kubernetes metadata
df.pod = df.ctx['pod']
df.service = df.ctx['service']
df.namespace = df.ctx['namespace']

# Filter to 5xx errors only
errors = df[df.resp_status >= 500]
errors = errors.groupby(['service', 'req_path']).agg(
  error_count=('resp_status', px.count)
)

# Latency percentiles across all requests
latency = df.groupby(['service']).agg(
  p50=('latency', px.quantiles(0.5)),
  p90=('latency', px.quantiles(0.9)),
  p99=('latency', px.quantiles(0.99)),
  throughput=('latency', px.count)
)

px.display(latency, 'HTTP Latency by Service')
px.display(errors, 'Error Count by Endpoint')

The killer feature: Pixie captures this data continuously with a ~2% CPU overhead per node, stores it in-memory (ring buffer), and you can query it retroactively. When an incident happens at 3am, you can query "what was the HTTP traffic pattern 5 minutes before the alert fired" — without any pre-configured dashboards or having instrumented your application.

5. Tetragon: Security Enforcement at Kernel Level

Tetragon (from Isovalent, Cilium's parent project) uses eBPF for real-time security enforcement. Unlike traditional audit tools that log events after they happen, Tetragon can block a syscall in-kernel before it completes. This is the difference between detecting a container escape and preventing it.

yaml — tetragon TracingPolicy: detect container escape attempts 
apiVersion: cilium.io/v1alpha1
kind: TracingPolicy
metadata:
  name: detect-nsenter
spec:
  kprobes:
    - call: "sys_setns"   # syscall used to switch namespaces (container escape technique)
      syscall: true
      args:
        - index: 0
          type: "int"    # file descriptor of target namespace
      selectors:
        - matchProcessNs:
            - namespace: "pid"
              values:
                - host_pid   # only alert if process is trying to join the host PID namespace
          matchActions:
            - action: Sigkill   # KILL the process immediately — not just log

---
# Also: alert on any write to /etc/passwd or /etc/shadow from a container
apiVersion: cilium.io/v1alpha1
kind: TracingPolicy
metadata:
  name: protect-passwd
spec:
  kprobes:
    - call: "fd_install"
      syscall: false
      args:
        - index: 1
          type: "file"
      selectors:
        - matchArgs:
            - index: 1
              operator: "Prefix"
              values:
                - "/etc/passwd"
                - "/etc/shadow"
          matchActions:
            - action: Sigkill

6. CPU Profiling with eBPF

Traditional CPU profilers require you to instrument the application or attach a profiler process that adds overhead. eBPF-based profiling uses perf events to sample the CPU stack at a configurable rate (typically 99 Hz) across all processes with minimal overhead — around 1% CPU.

bash — eBPF CPU profiling and flame graphs 
# Profile all CPU usage across the entire system for 30 seconds at 99Hz
# Uses bpftrace's perf event sampling
bpftrace -e '
profile:hz:99 {
  @[comm, kstack()] = count();
}
interval:s:30 {
  exit();
}' > stacks.txt

# Or use perf (with eBPF backend on modern kernels)
perf record -F 99 -a -g --call-graph=dwarf -- sleep 30
perf script | stackcollapse-perf.pl | flamegraph.pl > flame.svg

# Profile a specific container process (find its PID first)
CONTAINER_PID=$(docker inspect --format '{{.State.Pid}}' my-container)
bpftrace -e "profile:hz:99 /pid == ${CONTAINER_PID}/ { @[ustack()] = count(); }" \
  -c "sleep 30" | flamegraph.pl > container-flame.svg

# Find hottest kernel functions system-wide right now
bpftrace -e 'profile:hz:99 { @[kstack] = count(); } interval:s:10 { print(@); clear(@); exit(); }'

A practical use case: a Go service showed 40% CPU with no obvious cause. The flame graph from bpftrace showed 30% of CPU time in runtime.mallocgc — excessive heap allocation causing GC pressure. The fix was a sync.Pool for the hot allocation path, reducing CPU by 60%. Traditional APM showed "high CPU" but couldn't tell us why without adding pprof endpoints and coordinating a profiling window.

eBPF program limits in containers: Running eBPF tools inside Kubernetes pods requires CAP_BPF and CAP_SYS_ADMIN capabilities, and often host PID namespace access. For security reasons, run your observability agents (Pixie, Tetragon, Hubble) as DaemonSets with these privileges rather than giving arbitrary pods kernel access. Never run privileged: true on application containers just to enable eBPF tracing.

7. Detecting Container Escapes with Syscall Tracing

Container escapes typically follow a pattern: exploit a vulnerability to run code, then use kernel syscalls to break out of the namespace boundary. The most common techniques involve ptrace (to attach to a process outside the container), mount (to mount the host filesystem), or open with relative-to-root paths that cross namespace boundaries.

bash — bpftrace: detect suspicious syscall patterns 
# Detect any container process trying to use ptrace (common escape vector)
bpftrace -e '
tracepoint:syscalls:sys_enter_ptrace {
  if (args->request == PTRACE_ATTACH || args->request == PTRACE_SEIZE) {
    printf("ALERT: ptrace attach by PID %d (%s) targeting PID %d\n",
           pid, comm, args->pid);
  }
}'

# Detect unexpected mount syscalls from containers
bpftrace -e '
tracepoint:syscalls:sys_enter_mount {
  printf("mount: PID=%d COMM=%s source=%s target=%s\n",
         pid, comm, str(args->dev_name), str(args->dir_name));
}'

# Alert on any process writing to /proc/sysrq-trigger (common escape technique)
bpftrace -e '
tracepoint:syscalls:sys_enter_openat {
  if (str(args->filename) == "/proc/sysrq-trigger") {
    printf("CRITICAL: %s (PID %d) opening sysrq-trigger\n", comm, pid);
    signal("SIGKILL");
  }
}'

Summary

eBPF has moved from a niche kernel hacking tool to the foundation of production observability. Cilium and Hubble give you L7 network visibility without a service mesh. Pixie gives you automatic, retroactive application observability. Tetragon gives you runtime security enforcement that actually blocks attacks rather than logging them. And bpftrace gives you an interactive kernel debugging tool that can answer questions in 60 seconds that previously required a multi-day instrumentation effort.

Start with bpftrace — it has zero setup cost and zero production risk. Then deploy Pixie in a dev or staging cluster to see what automatic observability looks like. That usually converts the remaining skeptics.
