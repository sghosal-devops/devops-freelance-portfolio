Chaos Engineering with LitmusChaos: Breaking Things on Purpose

Souvik· March 2026· 10 min read

The most dangerous belief in software engineering is that your system is resilient because it hasn't failed yet. Absence of observed failure is not evidence of absence of failure modes. Chaos engineering is the discipline of proactively exposing those failure modes in a controlled way — before your users do it for you at the worst possible moment.

This is not about being reckless. This is about running experiments with clearly defined hypotheses, measurable steady-state baselines, and blast radius controls. Netflix calls them GameDays. Google SRE calls them DiRT (Disaster Recovery Testing) exercises. I call them "the reason I sleep at 3am instead of being paged."

1. The Principles Behind Chaos Engineering

The Principles of Chaos Engineering (principlesofchaos.org) define the practice precisely. The core idea: build a hypothesis about system behavior under turbulent conditions, design an experiment, run it with minimal blast radius, observe, and learn. The keyword is hypothesis. "Let's kill some pods and see what happens" is not chaos engineering — it's gambling. A proper experiment looks like this:

  • Steady state: p99 latency < 200ms, error rate < 0.1%, all pods Ready
  • Hypothesis: Killing one pod from a 3-replica Deployment will not change the steady state because the HPA and readiness probes will route traffic only to healthy pods
  • Experiment: Kill one random pod in the payment-api Deployment
  • Observation: Monitor p99 latency and error rate for 5 minutes post-kill
  • Learning: Either confirm the hypothesis or discover the gap

Start in staging. Always. Once you have a mature practice, you can move to production with careful guardrails — but staging must be treated as a production-equivalent environment (same resource sizes, same topology, real-ish traffic via synthetic load) or the experiments are meaningless.

2. LitmusChaos Architecture

LitmusChaos is a CNCF-incubating chaos engineering framework built for Kubernetes. Its architecture has three main components:

  • Chaos Operator: A Kubernetes operator that watches for ChaosEngine and ChaosExperiment CRDs and executes the corresponding experiment logic
  • ChaosHub: A GitOps-style catalog of pre-built experiments (pod-kill, node-drain, CPU hog, network latency, disk fill, and 50+ more). You point it at a Git repo containing experiment YAML definitions
  • ChaosCenter: The optional control plane UI that provides a workflow builder, scheduling, GitOps integration, and observability dashboards

For most teams, you only need the Chaos Operator + a local ChaosHub clone. ChaosCenter is valuable when multiple teams are running experiments and you need governance and audit trails.

bash — install-litmus.sh 
# Install LitmusChaos via Helm (operator only, no ChaosCenter)
helm repo add litmuschaos https://litmuschaos.github.io/litmus-helm/
helm repo update

helm install chaos litmuschaos/litmus \
  --namespace litmus \
  --create-namespace \
  --set portal.frontend.service.type=ClusterIP \
  --set adminConfig.SKIP_SSL_VERIFY=false

# Verify operator is running
kubectl get pods -n litmus
# NAME                                    READY   STATUS    RESTARTS
# chaos-litmus-frontend-xxx               1/1     Running   0
# chaos-litmus-server-xxx                 1/1     Running   0
# chaos-operator-ce-xxx                   1/1     Running   0

# Install generic ChaosExperiment definitions into your target namespace
kubectl apply -f https://hub.litmuschaos.io/api/chaos/master?file=charts/generic/experiments.yaml \
  -n production

3. Writing a ChaosEngine Manifest

The ChaosEngine is the central object. It binds a target application to one or more experiments, defines the steady-state hypothesis check (via probes), and controls execution parameters. Here's a production-realistic example for pod-kill with HTTP and Prometheus probes:

yaml — pod-kill-engine.yaml 
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosEngine
metadata:
  name: payment-api-pod-kill
  namespace: production
spec:
  appinfo:
    appns: production
    applabel: "app=payment-api"
    appkind: deployment

  engineState: "active"
  annotationCheck: "true"   # only targets pods with chaos annotation

  # Chaos Service Account with RBAC to delete pods
  chaosServiceAccount: litmus-admin

  experiments:
  - name: pod-delete
    spec:
      components:
        env:
        - name: TOTAL_CHAOS_DURATION
          value: "60"     # seconds the experiment runs
        - name: CHAOS_INTERVAL
          value: "15"     # kill a pod every 15 seconds
        - name: FORCE
          value: "false"  # graceful termination — use true only to simulate OOM kill
        - name: PODS_AFFECTED_PERC
          value: "33"     # affect max 33% of matching pods at once

      probes:
        # HTTP probe: verify the service endpoint stays healthy
      - name: payment-api-health
        type: httpProbe
        mode: Continuous
        runProperties:
          probeTimeout: 5000
          interval: 5
          retry: 2
          stopOnFailure: true   # abort experiment if probe fails — blast radius control
        httpProbe/inputs:
          url: "http://payment-api.production.svc.cluster.local/healthz"
          insecureSkipVerify: false
          method:
            get:
              criteria: ==
              responseCode: "200"

        # Prometheus probe: verify error rate stays below SLO
      - name: error-rate-slo
        type: promProbe
        mode: Continuous
        runProperties:
          probeTimeout: 10000
          interval: 10
          retry: 1
          stopOnFailure: true
        promProbe/inputs:
          endpoint: "http://prometheus.monitoring.svc.cluster.local:9090"
          query: |
            sum(rate(http_requests_total{job="payment-api",status=~"5.."}[2m]))
            / sum(rate(http_requests_total{job="payment-api"}[2m])) * 100
          comparator:
            criteria: "<="
            value: "1.0"    # SLO: error rate must stay below 1%
The annotation requirement: Notice annotationCheck: "true" — LitmusChaos will only target pods that have the annotation litmuschaos.io/chaos: "true" on their parent Deployment. This is a safety feature. Only explicitly annotated workloads are eligible for chaos experiments. Start by annotating only your staging deployments.

4. Network Latency Injection

Pod-kill experiments are table stakes. The more interesting experiments inject network faults that simulate real-world cloud networking issues: packet loss, latency spikes, bandwidth throttling. LitmusChaos's pod-network-latency experiment uses tc netem under the hood, running in a privileged init container to manipulate the pod's network namespace.

yaml — network-latency-engine.yaml 
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosEngine
metadata:
  name: api-network-latency
  namespace: production
spec:
  appinfo:
    appns: production
    applabel: "app=order-service"
    appkind: deployment
  engineState: "active"
  chaosServiceAccount: litmus-admin
  experiments:
  - name: pod-network-latency
    spec:
      components:
        env:
        - name: TOTAL_CHAOS_DURATION
          value: "120"
        - name: NETWORK_LATENCY
          value: "2000"      # 2000ms added latency — simulates degraded network
        - name: JITTER
          value: "200"       # +/- 200ms jitter
        - name: DESTINATION_IPS
          value: "10.0.5.20"  # target only the payment-gateway IP — not all traffic
        - name: DESTINATION_HOSTS
          value: "payment-gateway.production.svc.cluster.local"
        - name: CONTAINER_RUNTIME
          value: "containerd"
        - name: SOCKET_PATH
          value: "/run/containerd/containerd.sock"
        - name: PODS_AFFECTED_PERC
          value: "50"       # inject into 50% of matching pods

This experiment answers a critical question: "What happens to our order-service when the payment-gateway has a 2-second latency spike?" Does order-service time out gracefully? Does it queue requests? Does it cascade? Does the circuit breaker trip? You find out now, not during a real payment provider degradation event.

5. CPU Hog and Node Drain

The pod-cpu-hog experiment consumes CPU within target pods using a stress-ng process injected as a sidecar. This validates your HPA response time — does the horizontal pod autoscaler kick in before your latency SLO is breached? Typical HPA polling is every 15 seconds with a 3-minute stabilization window, which means you could have 3+ minutes of elevated CPU before scale-out helps. If your SLO is 200ms p99 and CPU saturation pushes you to 500ms within 30 seconds, you have a problem.

The node-drain experiment cordons a node and evicts all pods, simulating a node failure or a cloud provider maintenance event. This is the most important experiment for verifying PodDisruptionBudgets. If you have a 3-replica Deployment with a PDB requiring minAvailable: 2, does the drain actually respect that? Does your cluster have enough spare capacity to reschedule? Run this in staging regularly — I've seen teams discover their staging cluster was exactly at capacity and a node drain would have left services completely offline.

6. Integrating Chaos into CI/CD

Manual chaos experiments are valuable for GameDays, but the real value comes from automated chaos in your CD pipeline. The idea: after deploying to staging and running your integration test suite, run a short chaos experiment automatically and verify steady-state is maintained. If the experiment fails (probes breach), block the promotion to production.

yaml — chaos-ci-step.yaml 
# GitHub Actions step — runs after integration tests pass in staging
- name: Run Chaos Experiment
  run: |
    # Apply the ChaosEngine for a 60s pod-delete experiment
    kubectl apply -f chaos/pod-delete-engine.yaml

    # Wait for experiment to complete (poll ChaosResult)
    TIMEOUT=180
    ELAPSED=0
    while [ $ELAPSED -lt $TIMEOUT ]; do
      VERDICT=$(kubectl get chaosresult payment-api-pod-kill-pod-delete \
        -n production \
        -o jsonpath='{.status.experimentStatus.verdict}')

      if [ "$VERDICT" = "Pass" ]; then
        echo "Chaos experiment passed — system is resilient"
        kubectl delete -f chaos/pod-delete-engine.yaml
        exit 0
      elif [ "$VERDICT" = "Fail" ]; then
        echo "CHAOS EXPERIMENT FAILED — blocking promotion to production"
        kubectl delete -f chaos/pod-delete-engine.yaml
        exit 1
      fi

      echo "Experiment running... verdict: $VERDICT"
      sleep 10
      ELAPSED=$((ELAPSED + 10))
    done

    echo "Chaos experiment timed out"
    kubectl delete -f chaos/pod-delete-engine.yaml
    exit 1

7. Observability During Chaos: Prometheus Queries

During a chaos experiment, you need to watch three things in real time: error rate, latency percentiles, and pod readiness. Pin these queries in a Grafana dashboard dedicated to chaos experiments.

promql — chaos-dashboard-queries.txt 
# 1. HTTP Error Rate — should stay below 1% during pod-kill
sum(rate(http_requests_total{namespace="production",status=~"5.."}[1m]))
/ sum(rate(http_requests_total{namespace="production"}[1m])) * 100

# 2. p99 Latency — watching for latency spikes during experiment
histogram_quantile(0.99,
  sum(rate(http_request_duration_seconds_bucket{namespace="production"}[2m])) by (le, job)
)

# 3. Pod Ready Count — should recover to target replica count within 60s
sum(kube_pod_status_ready{namespace="production", condition="true"}) by (pod)

# 4. HPA desired vs current replicas — are we autoscaling correctly?
kube_horizontalpodautoscaler_status_desired_replicas{namespace="production"}
vs
kube_horizontalpodautoscaler_status_current_replicas{namespace="production"}

# 5. Chaos experiment duration marker — use annotations in Grafana
# Tag your Grafana dashboard with a time-range annotation when ChaosEngine status = active
kube_customresource_chaos_engine_status{engineState="active"}
Grafana annotations for chaos: Create a Grafana annotation data source backed by the LitmusChaos webhook. When a ChaosEngine goes active, post a Grafana annotation to your dashboard. This draws a vertical line on all your time-series graphs marking the exact start and end of the experiment — invaluable when reviewing whether the system recovered fast enough.

8. Building a Chaos Runbook

Every experiment needs a documented runbook before it runs. The runbook template I use:

  • Experiment ID: payment-api-pod-kill-v3
  • Date / Owner: who ran it and when
  • Steady State Definition: concrete metric values with Prometheus queries
  • Hypothesis: one sentence, falsifiable
  • Blast Radius: what can break (and what cannot)
  • Abort Criteria: probe conditions that will stop the experiment
  • Rollback Steps: if something goes wrong, how do you restore immediately
  • Observations: fill in post-experiment — what actually happened
  • Follow-Up Actions: if hypothesis was disproved, what gets fixed

Chaos engineering without documentation is just vandalism. The learning comes from the systematic recording of hypotheses, observations, and improvements. Track this in Confluence, Notion, or a Git repo — anywhere that accumulates institutional knowledge.

Never skip the abort criteria: The stopOnFailure: true setting on probes is your automated abort trigger. But also define a human abort procedure: who has the kubectl access to delete the ChaosEngine immediately, and what is the maximum duration anyone can wait before manually aborting if monitoring is unavailable. Production chaos experiments with no human oversight are reckless regardless of how good your probes are.
