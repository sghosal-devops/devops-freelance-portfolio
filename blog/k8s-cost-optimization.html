Kubernetes Cost Optimization: HPA, VPA, KEDA & Spot Instances

Souvik· April 2026· 13 min read

The first Kubernetes cost optimization engagement I did, the client was spending $180k/month on EKS. Four months later it was $74k/month — same workloads, same reliability SLAs, same team size. No magic, just systematic elimination of waste: right-sizing, autoscaling, Karpenter, and spot instances. This post is the exact playbook.

The most common mistakes I find: every pod is over-provisioned because teams are afraid to set tight limits, the cluster autoscaler is configured conservatively so nodes are half-empty, spot instances aren't used because of a 2019 horror story about a single instance type pool, and there's no visibility into what's actually costing money.

1. VPA in Recommendation Mode First — Never Blind Resize

Vertical Pod Autoscaler in Off or Auto mode without first understanding your workload's actual resource profile is how you cause OOMKills at 2am. The correct approach is a two-phase deployment: run VPA in recommendation-only mode for 2-4 weeks, collect data, review the recommendations, then apply them as static requests/limits updates.

yaml — vpa-recommendation.yaml 
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: api-vpa
  namespace: api-prod
spec:
  targetRef:
    apiVersion: "apps/v1"
    kind: Deployment
    name: api-server
  updatePolicy:
    updateMode: "Off"   # recommendation only, does NOT modify pods
  resourcePolicy:
    containerPolicies:
      - containerName: api
        minAllowed:
          cpu: 50m
          memory: 64Mi
        maxAllowed:
          cpu: 4
          memory: 8Gi
        controlledResources: ["cpu", "memory"]

After 2 weeks, check the recommendation:

bash — reading VPA recommendations 
kubectl describe vpa api-vpa -n api-prod

# Output (abbreviated):
# Recommendation:
#   Container Recommendations:
#     Container Name:  api
#     Lower Bound:     cpu: 120m, memory: 256Mi
#     Target:          cpu: 340m, memory: 512Mi  <-- use this for requests
#     Upper Bound:     cpu: 1200m, memory: 2Gi   <-- use this for limits
#     Uncapped Target: cpu: 340m, memory: 480Mi

# Compare to what you currently have set:
kubectl get deployment api-server -n api-prod \
  -o jsonpath='{.spec.template.spec.containers[0].resources}'

# Common finding: teams set cpu: 2 requests where VPA recommends 340m
# That's ~6x over-provisioning on CPU across every pod replica

In that $180k engagement, VPA recommendations showed an average of 4x over-provisioning on CPU across all workloads. Applying the recommendations and removing the padding engineers had added "just in case" cut the compute bill by 35% before we touched anything else.

VPA and HPA conflict. Do NOT run VPA in Auto mode simultaneously with HPA scaling on CPU/memory. When VPA resizes a pod, it evicts and recreates it — which changes the denominator for HPA's target utilization calculation, causing HPA to scale out, which causes VPA to try to right-size again. Use HPA for scaling (replicas) and VPA for sizing (requests/limits), but only one of them targeting the same metric at a time. If using both, configure HPA to scale on custom metrics (like RPS) rather than CPU.

2. HPA vs KEDA — Choosing the Right Scaler

HPA is the right choice when: your workload correlates directly with CPU or memory (compute-bound services, batch processors), and your scaling metric is naturally captured by the Kubernetes metrics server.

KEDA is the right choice when: your workload should scale on external signals — queue depth, Kafka consumer lag, HTTP requests per second from an external metrics source (Prometheus, DataDog), a cron schedule, or zero (KEDA can scale to zero, HPA cannot). For worker processes consuming from SQS or RabbitMQ, KEDA is strictly better.

yaml — KEDA ScaledObject: scale on SQS queue depth 
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: sqs-worker-scaler
  namespace: workers
spec:
  scaleTargetRef:
    name: sqs-worker
  minReplicaCount: 0    # scale to ZERO when queue is empty — huge cost saving
  maxReplicaCount: 50
  cooldownPeriod: 300   # wait 5min before scaling back to zero
  pollingInterval: 15   # check queue every 15 seconds
  triggers:
    - type: aws-sqs-queue
      authenticationRef:
        name: keda-aws-irsa
      metadata:
        queueURL: https://sqs.us-east-1.amazonaws.com/123456789/image-processing
        queueLength: "5"    # target: 5 messages per worker pod
        awsRegion: us-east-1
        identityOwner: operator   # use KEDA's IRSA role, not pod's role

A client had 10 always-on worker pods for an image processing pipeline. Peak load was during business hours (9am-6pm). Off-peak the queue was always empty. Switching to KEDA with scale-to-zero eliminated those 10 workers outside business hours — about 70% of the worker pool's monthly cost.

3. Karpenter vs Cluster Autoscaler: Karpenter Wins

Cluster Autoscaler (CAS) works on node groups. You define node groups in advance (m5.large spot, m5.xlarge on-demand, etc.) and CAS adds or removes nodes from those fixed groups. The problems: you must predict which instance types you need, scaling decisions are slow (2-3 minutes to provision a new node), and bin-packing is poor because CAS picks the cheapest node that fits the pending pod, not the globally optimal packing.

Karpenter eliminates node groups entirely. It watches for unschedulable pods and directly calls the EC2 API to provision exactly the right instance type for the pending workload — choosing from a fleet of allowed instance families. Provisioning is faster (under 60 seconds in most cases), bin-packing is better, and spot instance diversity is automatic.

yaml — karpenter NodePool and EC2NodeClass 
apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: general
spec:
  template:
    metadata:
      labels:
        node-pool: general
    spec:
      nodeClassRef:
        group: karpenter.k8s.aws
        kind: EC2NodeClass
        name: general
      requirements:
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["spot", "on-demand"]  # spot preferred, on-demand fallback
        - key: karpenter.k8s.aws/instance-category
          operator: In
          values: ["c", "m", "r"]
        - key: karpenter.k8s.aws/instance-generation
          operator: Gt
          values: ["5"]                   # only 6th gen and newer
        - key: kubernetes.io/arch
          operator: In
          values: ["amd64", "arm64"]   # Graviton eligible
  disruption:
    consolidationPolicy: WhenUnderutilized   # bin-pack and remove empty nodes
    consolidateAfter: 30s
  limits:
    cpu: 1000     # max 1000 vCPU across this pool
    memory: 4000Gi
---
apiVersion: karpenter.k8s.aws/v1
kind: EC2NodeClass
metadata:
  name: general
spec:
  amiFamily: AL2023
  role: KarpenterNodeRole-prod-us-east-1
  subnetSelectorTerms:
    - tags:
        karpenter.sh/discovery: prod-us-east-1
  securityGroupSelectorTerms:
    - tags:
        karpenter.sh/discovery: prod-us-east-1
  blockDeviceMappings:
    - deviceName: /dev/xvda
      ebs:
        volumeSize: 50Gi
        volumeType: gp3
        encrypted: true

By allowing Karpenter to pick from the full fleet of m6i, m6a, m6g, c6i, c6a, c6g instance types on spot, it can find spot capacity at a 70-90% discount to on-demand virtually always — because with 10+ instance types in the pool, the probability of all of them being capacity-constrained simultaneously is extremely low. This is the "spot instance diversity" rule that makes spot reliable in 2026.

Spot interruption handling. Karpenter responds to EC2 spot interruption notices (2-minute warning) by immediately cordoning the node and triggering pod eviction. Your pods must have PodDisruptionBudgets to prevent all replicas from being evicted simultaneously. For stateful workloads, use a spot interruption handler like the AWS Node Termination Handler (NTH) to gracefully drain in-flight requests before the node goes away.

4. Pod Topology Spread Constraints for Better Bin-Packing

Without topology constraints, the Kubernetes scheduler may place all replicas on a few large nodes, leaving other nodes underutilised. This wastes money and reduces resilience. Topology spread constraints force even distribution across nodes and AZs.

yaml — topology spread in Deployment spec 
spec:
  template:
    spec:
      topologySpreadConstraints:
        # Spread evenly across availability zones
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: api-server
        # Also spread across nodes (max 2 pods per node)
        - maxSkew: 1
          topologyKey: kubernetes.io/hostname
          whenUnsatisfiable: ScheduleAnyway   # soft constraint — prefer, don't require
          labelSelector:
            matchLabels:
              app: api-server
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 100
              podAffinityTerm:
                topologyKey: kubernetes.io/hostname
                labelSelector:
                  matchLabels:
                    app: api-server

5. Namespace Resource Quotas and LimitRanges

Without quotas, one team can accidentally (or intentionally) consume the entire cluster. LimitRanges set defaults so developers don't have to think about resource requests on every manifest — and containers without requests/limits are the single biggest source of resource waste because Kubernetes cannot schedule them efficiently.

yaml — namespace quotas and limit ranges 
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
  namespace: team-payments
spec:
  hard:
    requests.cpu: "20"       # sum of all pod CPU requests in namespace
    requests.memory: 40Gi
    limits.cpu: "40"
    limits.memory: 80Gi
    count/pods: "100"
    count/services: "20"
    count/persistentvolumeclaims: "20"
---
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: team-payments
spec:
  limits:
    - type: Container
      default:           # applied if container has no limits set
        cpu: 500m
        memory: 512Mi
      defaultRequest:    # applied if container has no requests set
        cpu: 100m
        memory: 128Mi
      max:               # container cannot request more than this
        cpu: "8"
        memory: 16Gi
      min:
        cpu: 10m
        memory: 16Mi

6. Kubecost for Cost Visibility

You cannot optimize what you cannot see. Kubecost allocates AWS/GCP/Azure costs down to the namespace, deployment, and label level. The sticker shock of seeing "this one batch job that runs once a week costs $8,000/month" is a powerful motivator for engineering teams to care about resource usage.

Install Kubecost via Helm, integrate it with your cloud provider's billing API (AWS Cost and Usage Reports for EKS), and create a dashboard that shows cost by team/namespace updated daily. We surface this as a weekly digest sent to each team's Slack channel: "your namespace spent $X this week, up/down Y% from last week." Teams respond to their own cost data in ways they never respond to platform team lectures about resource efficiency.

Real numbers from production: A 40-node EKS cluster at a Series B startup. Starting state: $52k/month. After VPA right-sizing (applied recommendations from 3-week observation): $38k/month (-27%). After switching worker pools to Karpenter + spot: $24k/month (-37% on top of that). After KEDA scale-to-zero on batch workers and reducing off-hours baseline: $18k/month. Final run rate: $18k vs $52k starting point — 65% reduction over 6 months, with the same workloads and improved reliability due to multi-AZ spot diversity.

7. Additional Wins: PVC Right-Sizing and EBS gp2 to gp3 Migration

Two quick wins that are often overlooked: storage costs can be 10-15% of a Kubernetes cluster's cloud bill, and most of it is waste.

First, find all EBS volumes still on gp2. A simple kubectl get pvc -A combined with checking the StorageClass will reveal them. Migrating gp2 to gp3 is a 20% cost reduction with better baseline performance — and you can do it online without detaching the volume. Second, find PVCs that are 80%+ empty. This requires Kubecost or querying the kubelet_volume_stats_used_bytes metric from Prometheus: divide it by kubelet_volume_stats_capacity_bytes and find volumes under 20% utilization. For many teams, databases were provisioned at 500Gi "to be safe" and are using 30Gi.
