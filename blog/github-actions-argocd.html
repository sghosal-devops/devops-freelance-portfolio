Production GitOps: GitHub Actions + ArgoCD End-to-End

Souvik· May 2026· 14 min read

GitOps is not just "store your YAML in Git." Done right, it means the Git repository is the single source of truth for cluster state, and the cluster enforces that truth continuously. Every change — planned or emergency — goes through a PR. ArgoCD watches the manifest repo and reconciles the cluster to match. GitHub Actions builds, tests, scans, and pushes the image and manifest updates. Nobody kubectl applys directly to production.

This post walks through the complete production pipeline I've deployed for multiple clients, with the actual YAML that makes it work.

1. The Full Pipeline Flow

Here's the end-to-end path from a developer's git push to a running pod in production:

  1. Developer pushes to a feature branch → opens PR to main
  2. GitHub Actions runs: unit tests → docker build → Trivy image scan → push to ECR/GHCR with commit SHA tag
  3. On merge to main: GHA updates the image tag in the manifest repo via a PR (or direct push to a branch)
  4. Manifest repo PR is reviewed and merged (or auto-merged for non-prod)
  5. ArgoCD detects the manifest change and syncs the cluster
  6. ArgoCD sync hooks run database migrations, smoke tests, etc.
  7. If health checks fail, ArgoCD stops the sync and alerts

The critical separation is between the application repo (source code, Dockerfile, GHA workflows) and the manifest repo (Kubernetes YAML, Helm values, ArgoCD applications). This separation is what makes GitOps auditable — you can look at the manifest repo's git log and know exactly what was deployed, when, by whom, and via which CI run.

2. GitHub Actions: Build, Scan, and Push

yaml — .github/workflows/ci.yaml 
name: CI — Build, Scan, Push

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build-scan-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
      security-events: write   # for SARIF upload
      id-token: write           # for OIDC signing

    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=sha,format=long
            type=ref,event=branch
            type=semver,pattern={{version}}

      - name: Build and push image
        id: build
        uses: docker/build-push-action@v5
        with:
          context: .
          push: ${{ github.event_name != 'pull_request' }}
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

      - name: Trivy vulnerability scan
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:sha-${{ github.sha }}
          format: sarif
          output: trivy-results.sarif
          severity: CRITICAL,HIGH
          exit-code: '1'   # fail the build on CRITICAL/HIGH CVEs
          ignore-unfixed: true

      - name: Upload Trivy results to GitHub Security
        uses: github/codeql-action/upload-sarif@v3
        if: always()   # upload even if scan fails, so you see what's there
        with:
          sarif_file: trivy-results.sarif

      - name: Update manifest repo
        if: github.ref == 'refs/heads/main'
        env:
          GH_TOKEN: ${{ secrets.MANIFEST_REPO_PAT }}
          IMAGE_TAG: sha-${{ github.sha }}
        run: |
          git clone https://x-access-token:${GH_TOKEN}@github.com/myorg/k8s-manifests.git
          cd k8s-manifests
          yq e ".image.tag = \"${IMAGE_TAG}\"" -i apps/api/values-prod.yaml
          git config user.email "ci@myorg.com"
          git config user.name "CI Bot"
          git add apps/api/values-prod.yaml
          git commit -m "chore: update api to ${IMAGE_TAG}"
          git push
Use yq not sed. Updating image tags with sed -i 's/tag:.*/tag: sha-abc/' breaks on the first YAML comment, multi-line value, or nested key. yq parses the YAML correctly. Pin its version in CI and add a checksum verification if you care about supply-chain security.

3. ArgoCD Application Manifests

The ArgoCD Application CR is what connects the manifest repo to the cluster. Each application points to a path in Git and a destination namespace in the cluster.

yaml — argocd/apps/api.yaml 
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: api-prod
  namespace: argocd
  annotations:
    notifications.argoproj.io/subscribe.on-sync-failed.slack: platform-alerts
    notifications.argoproj.io/subscribe.on-health-degraded.slack: platform-alerts
spec:
  project: production
  source:
    repoURL: https://github.com/myorg/k8s-manifests.git
    targetRevision: HEAD
    path: apps/api
    helm:
      valueFiles:
        - values-base.yaml
        - values-prod.yaml
  destination:
    server: https://kubernetes.default.svc
    namespace: api-prod
  syncPolicy:
    automated:
      prune: true      # delete resources removed from Git
      selfHeal: true   # revert manual kubectl changes
    syncOptions:
      - CreateNamespace=true
      - PrunePropagationPolicy=foreground
      - RespectIgnoreDifferences=true
    retry:
      limit: 3
      backoff:
        duration: 10s
        factor: 2
        maxDuration: 2m
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas   # HPA manages this, ignore drift

4. App of Apps Pattern

Managing 30 ArgoCD Application manifests individually gets unwieldy fast. The App of Apps pattern solves this: one ArgoCD Application points to a directory of other ArgoCD Application manifests. ArgoCD deploys the parent app, which discovers and deploys all child apps.

yaml — argocd/root-app.yaml 
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: root-app
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/k8s-manifests.git
    targetRevision: HEAD
    path: argocd/apps   # directory containing all Application manifests
  destination:
    server: https://kubernetes.default.svc
    namespace: argocd
  syncPolicy:
    automated:
      prune: true
      selfHeal: true

Bootstrap by applying root-app.yaml once: kubectl apply -f argocd/root-app.yaml. From that point on, adding a new application is just adding a new YAML file to argocd/apps/ and merging the PR. ArgoCD discovers it on the next sync cycle.

5. Sync Waves and Hooks

Not all resources should sync at the same time. Database migrations must complete before the new application version starts. CRDs must exist before CR instances. Sync waves control the ordering.

yaml — apps/api/templates/db-migrate-job.yaml 
apiVersion: batch/v1
kind: Job
metadata:
  name: api-db-migrate-{{ .Release.Revision }}
  annotations:
    argocd.argoproj.io/hook: PreSync           # runs before any resources sync
    argocd.argoproj.io/hook-delete-policy: HookSucceeded
    argocd.argoproj.io/sync-wave: "-1"          # lower number = earlier wave
spec:
  ttlSecondsAfterFinished: 3600
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: {{ .Values.image.repository }}:{{ .Values.image.tag }}
          command: ["python", "manage.py", "migrate", "--noinput"]
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: api-secrets
                  key: database-url

Wave numbering: -2 for CRD installations, -1 for migrations and pre-flight checks, 0 (default) for the main application, 1 for post-deploy smoke tests. The PostSync hook is useful for sending a deployment notification to Slack or DataDog.

6. Secrets Management with External Secrets Operator

Never store plaintext secrets in the manifest repo. Sealed Secrets encrypts secrets at rest in Git (encrypted with the cluster's public key), but the ESO (External Secrets Operator) approach is more flexible: secrets live in AWS Secrets Manager or Vault and ESO syncs them into Kubernetes Secrets on a schedule.

yaml — apps/api/templates/external-secret.yaml 
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: api-secrets
  namespace: api-prod
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets-manager
    kind: ClusterSecretStore
  target:
    name: api-secrets
    creationPolicy: Owner
  data:
    - secretKey: database-url
      remoteRef:
        key: /prod/api/database-url
    - secretKey: jwt-secret
      remoteRef:
        key: /prod/api/jwt-secret
    - secretKey: stripe-key
      remoteRef:
        key: /prod/api/stripe-key
ESO vs Sealed Secrets: Sealed Secrets is simpler to operate (no external dependency) but ties secrets to a specific cluster. If you rotate the sealing key or rebuild the cluster, you must re-encrypt all secrets. ESO is more operationally complex but stores secrets in a system you already manage (Vault, AWS SM) and works across clusters. For multi-cluster or multi-cloud setups, ESO wins. For a single cluster with a small team, Sealed Secrets is fine.

7. ArgoCD RBAC

ArgoCD has its own RBAC model layered on top of Kubernetes. By default, anyone authenticated can see all applications — which in a production cluster is too permissive. Define ArgoCD Projects to scope what each team can do.

yaml — argocd/projects/production.yaml 
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: production
  namespace: argocd
spec:
  description: Production workloads — platform team gated
  sourceRepos:
    - https://github.com/myorg/k8s-manifests.git
  destinations:
    - namespace: api-prod
      server: https://kubernetes.default.svc
    - namespace: payments-prod
      server: https://kubernetes.default.svc
  clusterResourceWhitelist:
    - group: ''
      kind: Namespace
  namespaceResourceBlacklist:
    - group: ''
      kind: ResourceQuota   # apps can't modify their own quotas
  roles:
    - name: read-only
      description: View-only access for developers
      policies:
        - p, proj:production:read-only, applications, get, production/*, allow
      groups:
        - myorg:developers
    - name: deployer
      description: Can sync but not delete apps
      policies:
        - p, proj:production:deployer, applications, sync, production/*, allow
      groups:
        - myorg:platform-team

8. Rollback Strategy

ArgoCD's rollback is to pin the application to a previous git commit, not to use kubectl rollout undo. This is the key GitOps principle: Git is the source of truth, so rolling back means reverting in Git.

Operationally, the fastest rollback is: identify the last known-good commit SHA in the manifest repo, git revert (not reset) the bad commit, merge the revert PR, and let ArgoCD sync. For true emergencies where a PR is too slow, ArgoCD UI has a "Rollback" button that pins the application to a previous History state — but this puts the app into an "out-of-sync" state that must be reconciled by a Git revert afterwards.

ArgoCD Image Updater can automate image tag updates without needing GHA to write to the manifest repo. It watches your container registry for new tags matching a pattern and commits the update directly. For rapid iteration in dev/staging this is useful. For production, prefer the explicit GHA commit approach — it gives you a PR and a human review step before the image hits prod.

Summary

A mature GitOps setup takes about two weeks to build properly and pays dividends for years. The critical elements: strict separation of app repo and manifest repo, Trivy scanning that fails the build on unpatched CRITICAL CVEs, ArgoCD App of Apps for scalable application management, sync waves for migration ordering, ESO for secrets, and ArgoCD RBAC aligned with your team structure. Once this is running, every deployment is a git commit, every rollback is a git revert, and your audit trail is complete.
